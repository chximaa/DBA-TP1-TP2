Lab 2 — Instance Management, CDB/PDB Architecture

Starting instances · SPFILE/PFILE · CDB$ROOT · Pluggable Databases

Exercise 1

Starting an Instance

Step 1
Connect to the database as the user SYS.
SQL*PlusTerminal / CMD
-- Option A: From the OS command line
sqlplus / as sysdba

-- Option B: With explicit password
sqlplus sys/your_password as sysdba
Why AS SYSDBA?
SYS is the most powerful Oracle user. To use it, you must specify the role AS SYSDBA. Without it, Oracle will refuse the connection. SYSDBA gives you full admin rights — including the ability to start and stop the database.
SYS vs SYSTEM:
  • SYS → owns the data dictionary, must connect AS SYSDBA or AS SYSOPER
  • SYSTEM → normal DBA user, no special role required
  • The / as sysdba syntax uses OS authentication (no password needed if you are in the dba OS group)
Step 2
Display the SID of the instance.
SQLv$instance view
SELECT instance_name FROM v$instance;

-- Alternative using SHOW PARAMETER:
SHOW PARAMETER instance_name;
What is a SID? SID = System Identifier. It is the unique name for a running Oracle instance on the machine. v$instance is a dynamic performance view that provides real-time information about the current instance. The column instance_name contains the SID value (e.g., orcl).
Step 3
Display the metadata (name, type, path) related to the SPFILE.
SQLv$parameter view
SELECT name, type, value
FROM   v$parameter
WHERE  name = 'spfile';

-- Or even simpler:
SHOW PARAMETER spfile;
What is an SPFILE? The SPFILE (Server Parameter File) is a binary file that Oracle reads at startup to know all configuration parameters (memory sizes, paths, etc.). It is stored on the server. The query above returns its full path on disk.
SPFILE vs PFILE:
  • SPFILE → binary, managed by Oracle, modified with ALTER SYSTEM, default since Oracle 9i
  • PFILE (initSID.ora) → text file, edited manually in a text editor, used for troubleshooting
  • Oracle prefers SPFILE. If both exist, SPFILE takes priority at startup.
Step 4
Display the startup time of the instance.
SQLv$instance view
SELECT startup_time
FROM   v$instance;
startup_time is a column in v$instance that records the exact date/time when the current instance was last started. Useful for auditing and uptime checks.
Step 5
Shut down the database.
SQLShutdown modes
-- Most commonly used in labs — safe and clean:
SHUTDOWN IMMEDIATE;

-- Other modes (for reference):
SHUTDOWN NORMAL;     -- waits for all sessions to disconnect
SHUTDOWN TRANSACTIONAL; -- waits for transactions to complete
SHUTDOWN ABORT;      -- force-kills (like a power cut, avoid in production)
Shutdown Modes — must know for the exam:
ModeWaits for sessions?Waits for transactions?Recovery needed?
NORMALYESYESNO
TRANSACTIONALNOYESNO
IMMEDIATENONONO
ABORTNONOYES
Step 6
Create a PFILE from the SPFILE. Place it in path/orcl/PFILE/ named init1.ora.
SQLMust be connected as SYS (database can be in NOMOUNT or MOUNT state)
-- Step 6a: Create the PFILE from the existing SPFILE
CREATE PFILE='path/orcl/PFILE/init1.ora'
  FROM SPFILE='C:\app\yourNAME\product\12.1.0\dbhome_1\database\SPFILEorcl.ora';

-- If you want Oracle to find the SPFILE automatically (uses default location):
CREATE PFILE='path/orcl/PFILE/init1.ora' FROM SPFILE;
What is happening here? You are converting the binary SPFILE into a readable text file (PFILE). This is useful when you want to manually edit a parameter that cannot be changed dynamically, or when you need to troubleshoot a startup failure.
The database must be SHUT DOWN before creating a PFILE to use for startup.

You cannot start the database with a PFILE while the instance is already running. The CREATE PFILE command itself can run while the DB is open, but to then USE the PFILE to start, the DB must first be down.

Step 7
Start the database using the PFILE (you must be connected as "sys").
SQLStartup with explicit PFILE
STARTUP PFILE='path/orcl/PFILE/init1.ora';
Without specifying PFILE: STARTUP alone will look for the SPFILE in default locations. By specifying PFILE=, you force Oracle to use your text init file instead of the binary SPFILE.
Step 8
Display the sessions connected to the database using the v$session view.
SQLv$session view
-- Full list of all sessions:
SELECT * FROM v$session;

-- Cleaner, more useful version:
SELECT sid, serial#, username, status, machine, program
FROM  v$session
WHERE username IS NOT NULL;

-- Count total active sessions:
SELECT COUNT(*) FROM v$session;
v$session shows all current sessions (connections) to the database. Important columns: SID (session ID), USERNAME (who is connected), STATUS (ACTIVE or INACTIVE), MACHINE (client machine name). Background processes (like SMON, PMON) have NULL username.
Exercise 2

Information about the CDB and PDBs

Essential background: What is CDB/PDB architecture?
  • CDB (Container Database): The main Oracle database container. It has one root container called CDB$ROOT.
  • PDB (Pluggable Database): A self-contained database that lives inside the CDB. Like a tenant in an apartment building.
  • CDB$ROOT: The root of the CDB. When you connect with SYSDBA, you start here.
  • PDB$SEED: A read-only template used to create new PDBs. You cannot modify it.
Steps 1 & 2
1. Display the name of the instance.   2. Give the name of the current CDB.
SQL
-- Step 1: Instance name (the running Oracle process)
SELECT instance_name FROM v$instance;
-- Or:
SHOW con_name;   -- shows current container context

-- Step 2: Name of the current CDB
SELECT name, cdb FROM v$database;
-- Column 'cdb' will show YES if this is a CDB, NO if it is not
Instance name vs Database name: The instance is the memory + processes running. The database is the actual data files. They often have the same name, but they are different concepts. v$instance.instance_name → instance. v$database.name → database.
Step 3
Display the list of PDBs contained within the CDB$ROOT container.
SQLMust be connected to CDB$ROOT as SYSDBA
-- Method 1: using v$pdbs view
SELECT name, con_id, open_mode
FROM  v$pdbs;

-- Method 2: SQL*Plus shortcut
SHOW PDBS;
v$pdbs shows all PDBs known to the CDB. You will always see at least PDB$SEED (the template, always MOUNTED/READ ONLY) plus any PDBs you have created.
Step 4
Display all PDBs with their current startup states.
SQL
SELECT name, open_mode, restricted
FROM  v$pdbs;
PDB Open Modes — know these!
open_modeMeaning
MOUNTEDPDB is closed (not accessible to users)
READ WRITEPDB is open — users can read AND write data
READ ONLYPDB is open but no data modifications allowed
MIGRATEPDB is in upgrade mode
Step 5
Connect to the PDB named pdborcl, then display the name of the current PDB.
SQL
-- Connect to the PDB (switching container context)
ALTER SESSION SET CONTAINER = pdborcl;

-- Verify you are now inside pdborcl
SHOW CON_NAME;
-- Should output: CON_NAME = PDBORCL

-- Alternative: check v$pdbs (from CDB$ROOT)
SELECT sys_context('USERENV', 'CON_NAME') FROM dual;
ALTER SESSION SET CONTAINER does not change the connection — you stay connected as SYSDBA. But it switches your context from CDB$ROOT into the PDB. Think of it like entering a room inside a building.
Step 6
Display the open mode of "pdborcl", then activate read/write mode.
SQL
-- From CDB$ROOT, check current mode:
SELECT name, open_mode FROM v$pdbs WHERE name = 'PDBORCL';

-- Activate read/write mode:
ALTER PLUGGABLE DATABASE pdborcl OPEN READ WRITE;
Step 7
Shut down the database, then open it in read-only mode.
SQLRun from CDB$ROOT as SYSDBA
-- Close the PDB first
ALTER PLUGGABLE DATABASE pdborcl CLOSE IMMEDIATE;

-- Reopen in READ ONLY mode
ALTER PLUGGABLE DATABASE pdborcl OPEN READ ONLY;
Step 8
Connect as HR and insert a row into REGIONS. What happens?
SQL
-- Connect as HR user inside pdborcl
conn hr/hr@pdborcl

-- Try to insert:
INSERT INTO regions VALUES (5, 'Mars');
What happens? You will get an error like:
ORA-16000: database or pluggable database open for read-only access

Why? Because in Step 7 we opened pdborcl in READ ONLY mode. In this mode, no INSERT, UPDATE, or DELETE operations are allowed — only SELECT queries work. This is by design — it protects the data from accidental modification.
Step 9
Reassign read-write mode, then insert the row.
SQL
-- First, reconnect as SYSDBA to change the PDB mode
conn sys/password@orcl as sysdba

-- Close and reopen in READ WRITE mode
ALTER PLUGGABLE DATABASE pdborcl CLOSE IMMEDIATE;
ALTER PLUGGABLE DATABASE pdborcl OPEN READ WRITE;

-- Now reconnect as HR and insert
conn hr/hr@pdborcl
INSERT INTO regions VALUES (5, 'Mars');
COMMIT;
This time the INSERT will succeed because the PDB is back in READ WRITE mode. Don't forget COMMIT — in Oracle, DML (INSERT/UPDATE/DELETE) is not automatically committed.
Step 10
Activate the session in restricted mode.
SQL
-- Must be connected to the PDB as SYSDBA
ALTER PLUGGABLE DATABASE pdborcl RESTRICTED;

-- Verify:
SELECT name, open_mode, restricted FROM v$pdbs WHERE name = 'PDBORCL';
RESTRICTED mode means only users with the RESTRICTED SESSION system privilege can connect. Regular users like HR are blocked. This is used for maintenance tasks (schema changes, imports). It is NOT the same as READ ONLY — in restricted mode you can still write data, you just limit who can connect.
Step 11
Connect as user HR with password HR. What do you notice?
What happens: The connection is refused.

Oracle returns: ORA-01035: ORACLE only available to users with RESTRICTED SESSION privilege

Why? In RESTRICTED mode, only users who explicitly have the RESTRICTED SESSION privilege can log in. HR is a regular user without this privilege, so Oracle blocks the connection entirely. Only SYSDBA-level accounts can access the database in this state.
Exercise 3

Creating a PDB

Before starting — Prerequisites to create a PDB:
  • The CDB must be in READ/WRITE mode (not READ ONLY)
  • You must have the CREATE PLUGGABLE DATABASE system privilege
  • You must choose a unique name for the PDB within the CDB
Steps 1 → 3
1. Use DBCA to create pdbCS.   2. Shutdown + startup.   3. Show pdbs.
Step 1 (DBCA): DBCA (Database Configuration Assistant) is Oracle's GUI tool. Open it → choose "Manage Pluggable Databases" → Create PDB → follow wizard → name it pdbCS.

Alternatively, via command line:
SQLCommand-line alternative to DBCA
-- Create pdbCS from the seed (PDB$SEED template)
CREATE PLUGGABLE DATABASE pdbCS
  ADMIN USER pdbCS_admin IDENTIFIED BY password123
  FILE_NAME_CONVERT = ('C:\app\yourNAME\oradata\orcl\pdbseed\',
                       'C:\app\yourNAME\oradata\orcl\pdbCS\');
SQLStep 2 & 3
-- Step 2: Shutdown and restart the CDB
SHUTDOWN IMMEDIATE;
STARTUP;

-- Step 3: Show all PDBs
SHOW PDBS;
-- or:
SELECT name, open_mode FROM v$pdbs;
After CDB restart, PDBs start in MOUNTED state by default!

Even if pdbCS was open before the shutdown, Oracle does NOT automatically reopen PDBs after a CDB restart unless you configure auto-start triggers. You must manually open each PDB you need.

Step 4
Start the pdbCS database using ALTER PLUGGABLE DATABASE.
SQL
ALTER PLUGGABLE DATABASE pdbCS OPEN READ WRITE;
Steps 5 & 6
5. Display current PDB status.   6. Open ALL PDBs simultaneously.
SQL
-- Step 5: Check all PDB statuses
SELECT name, open_mode FROM v$pdbs;

-- Step 6: Open ALL PDBs at once using the ALL option
ALTER PLUGGABLE DATABASE ALL OPEN READ WRITE;
The ALL keyword is a very convenient shortcut — it targets every PDB in the CDB simultaneously. Without it, you would need a separate ALTER command for each PDB.
Steps 7, 8 & 9
STARTUP command, open pdbCS, display status.
SQL
-- Step 7: STARTUP can also open a PDB (when connected inside it):
ALTER SESSION SET CONTAINER = pdbCS;
STARTUP;     -- opens the current PDB only

-- Check v$pdbs from CDB$ROOT
ALTER SESSION SET CONTAINER = CDB$ROOT;
SELECT name, open_mode FROM v$pdbs;

-- Step 8: Simple open (defaults to READ WRITE if not specified)
ALTER PLUGGABLE DATABASE pdbCS OPEN;

-- Step 9: Display status again
SHOW PDBS;
Steps 10 & 11
10. Close pdbCS immediately.   11. Close ALL PDBs EXCEPT pdbCS.
SQL
-- Step 10: Close one specific PDB
ALTER PLUGGABLE DATABASE pdbCS CLOSE IMMEDIATE;

-- Step 11: Close ALL PDBs EXCEPT pdbCS
ALTER PLUGGABLE DATABASE ALL EXCEPT pdbCS CLOSE IMMEDIATE;
Key syntax summary for the exam:
GoalCommand
Open one PDB (RW)ALTER PLUGGABLE DATABASE pdbCS OPEN READ WRITE;
Open one PDB (default)ALTER PLUGGABLE DATABASE pdbCS OPEN;
Open ALL PDBsALTER PLUGGABLE DATABASE ALL OPEN READ WRITE;
Close one PDBALTER PLUGGABLE DATABASE pdbCS CLOSE IMMEDIATE;
Close ALL EXCEPT oneALTER PLUGGABLE DATABASE ALL EXCEPT pdbCS CLOSE IMMEDIATE;
Exercise 4

Direct Connection to PDB without going through the CDB

Why does this need special setup?

So far we connected to PDBs by first connecting to the CDB, then switching container (ALTER SESSION SET CONTAINER). But in production, applications need to connect directly to a PDB using a connection string like @pdbCS. For this to work, Oracle's listener must know about the PDB — configured in tnsnames.ora.

Steps 1, 2 & 3
1. Verify listener is running.   2. Try to connect directly to pdbCS.   3. What happens?
OS Terminal / CMD
-- Step 1: Check listener status
lsnrctl status

-- Step 2: Try connecting directly to pdbCS
sqlplus sys/your_password@pdbCS AS SYSDBA
Step 3 — What happens?
The connection FAILS with:
ORA-12154: TNS:could not resolve the connect identifier specified

Why? Oracle uses the file tnsnames.ora to resolve names like @pdbCS to a real network address (host + port + service). Since we haven't added pdbCS to that file yet, Oracle doesn't know what pdbCS means as a network destination.
Steps 4 & 5
Open tnsnames.ora and add an alias entry for pdbCS.
Where is tnsnames.ora? Default path:
$ORACLE_HOME/network/admin/tnsnames.ora
e.g.: C:\app\yourNAME\product\12.1.0\dbhome_1\network\admin\tnsnames.ora

Open it with a text editor (Notepad on Windows, gedit/nano on Linux).
tnsnames.oraAdd this block to the file
PDBCS =
  (DESCRIPTION =
    (ADDRESS = (PROTOCOL = TCP)(HOST = localhost)(PORT = 1521))
    (CONNECT_DATA =
      (SERVER = DEDICATED)
      (SERVICE_NAME = pdbCS)
    )
  )
What each part means:
• PDBCS → the alias name you will type after the @
• HOST = localhost → the server where Oracle runs (localhost = this machine)
• PORT = 1521 → Oracle's default listener port
• SERVICE_NAME = pdbCS → the Oracle service name for this PDB (usually same as PDB name)
Steps 6 & 7
6. Connect using the same command from Step 2.   7. Display the current PDB.
SQL
-- Step 6: Now the direct connection works
sqlplus sys/your_password@pdbCS AS SYSDBA

-- Step 7: Confirm you are inside pdbCS
SHOW CON_NAME;
-- Output: CON_NAME = PDBCS
tnsnames.ora — exam key points:
  • Located in $ORACLE_HOME/network/admin/
  • Maps aliases to network addresses — needed for remote/direct PDB connections
  • After editing, no restart needed — Oracle reads it fresh on each new connection
  • Use tnsping pdbCS in the terminal to test if the alias resolves correctly

Lab 3 — Initialization Parameters & Dynamic Performance Views

SPFILE scopes · Parameter management · v$ views for monitoring

Exercise 1

Initialization Parameters

The 3 parameter views — crucial to understand:
ViewWhat it showsWhen it updates
v$parameter Current values in memory (what the running instance uses) Immediately when SCOPE=MEMORY or SCOPE=BOTH
v$spparameter Values stored in the SPFILE on disk Immediately when SCOPE=SPFILE or SCOPE=BOTH
v$system_parameter Current system-level values in memory (similar to v$parameter for system-wide params) Immediately when changed at SYSTEM level
Step 1
Display the spfile location.
SQL
-- Method 1
SELECT value FROM v$parameter WHERE name = 'spfile';

-- Method 2 (simpler)
SHOW PARAMETER spfile;
If the value is empty (NULL), Oracle is currently running with a PFILE (text init file), not an SPFILE. If you see a path, Oracle is using an SPFILE located at that path.
Step 2
Display the maximum number of connected processes from both v$parameter and v$spparameter.
SQL
-- 2a: From v$parameter (in-memory value)
SELECT name, value, description
FROM  v$parameter
WHERE name = 'processes';

-- 2b: From v$spparameter (value stored in SPFILE)
SELECT name, value, isspecified
FROM  v$spparameter
WHERE name = 'processes';
processes controls the maximum number of OS processes that can connect to Oracle simultaneously. The default is typically 150–300. This is a static parameter — changes require a database restart.

isspecified in v$spparameter: TRUE = explicitly set in SPFILE, FALSE = using the default value.
Step 3
Modify processes in the SPFILE using SCOPE=SPFILE. Query both views. What do you notice?
SQL
-- Modify the value in the SPFILE only
ALTER SYSTEM SET processes = 300 SCOPE=SPFILE;

-- Now check both views:
SELECT name, value FROM v$parameter   WHERE name = 'processes';
SELECT name, value FROM v$spparameter WHERE name = 'processes';
What you observe after SCOPE=SPFILE:
  • v$spparameter → shows the NEW value (300) — because the SPFILE on disk was updated
  • v$parameter → shows the OLD value (e.g., 150) — the running instance has NOT changed yet
The change is "pending" — it will take effect only after the next database restart.
SCOPE values — this is a very common exam topic:
SCOPEUpdates SPFILE?Updates Memory?Survives restart?Typical use
SPFILE YESNOYES Static params that need restart
MEMORY NOYESNO Temporary / test changes
BOTH YESYESYES Dynamic params, permanent change

Default when SCOPE is not specified: BOTH (if dynamic) or SPFILE (if static).

Step 4
Stop and restart the instance, then re-display the values. What do you notice?
SQL
SHUTDOWN IMMEDIATE;
STARTUP;

-- Check both views again
SELECT name, value FROM v$parameter   WHERE name = 'processes';
SELECT name, value FROM v$spparameter WHERE name = 'processes';
After restart: Both views now show 300. The instance started by reading the SPFILE, which had the new value. So memory is now in sync with the SPFILE.
Step 5
Display the maximum number of open cursors from all 3 views.
SQL
-- From all 3 views simultaneously
SELECT 'v$parameter'        source, name, value FROM v$parameter        WHERE name='open_cursors'
UNION ALL
SELECT 'v$spparameter'      source, name, value FROM v$spparameter      WHERE name='open_cursors'
UNION ALL
SELECT 'v$system_parameter' source, name, value FROM v$system_parameter WHERE name='open_cursors';
open_cursors = maximum number of open cursor handles a user can have simultaneously. Default is 300. Unlike processes, this is a dynamic parameter — it can be changed without restart using SCOPE=MEMORY or SCOPE=BOTH.
Step 6
Modify open_cursors IN MEMORY using SCOPE=MEMORY. Check all 3 views. What do you notice?
SQL
ALTER SYSTEM SET open_cursors = 400 SCOPE=MEMORY;

-- Check all 3 views:
SELECT name, value FROM v$parameter        WHERE name='open_cursors';
SELECT name, value FROM v$spparameter      WHERE name='open_cursors';
SELECT name, value FROM v$system_parameter WHERE name='open_cursors';
What you observe after SCOPE=MEMORY:
  • v$parameter → shows 400 (new value in memory)
  • v$system_parameter → shows 400 (new value in memory)
  • v$spparameter → shows OLD value (e.g., 300) — the SPFILE was NOT touched
The change is live NOW, but it will NOT survive a restart.
Step 7
Stop and restart, then re-display the values from all 3 views. What do you notice?
SQL
SHUTDOWN IMMEDIATE;
STARTUP;

SELECT name, value FROM v$parameter        WHERE name='open_cursors';
SELECT name, value FROM v$spparameter      WHERE name='open_cursors';
SELECT name, value FROM v$system_parameter WHERE name='open_cursors';
After restart: All 3 views show the OLD value again (e.g., 300). The MEMORY change was lost because it was never written to the SPFILE. Oracle reloads parameters from the SPFILE at startup.
Step 8
Try modifying other parameters using the different scope options.
SQLExamples with different scopes
-- Example: change SGA target using SCOPE=BOTH (if it's dynamic)
ALTER SYSTEM SET sga_target = 600M SCOPE=BOTH;

-- Example: test SCOPE=SPFILE only (static param)
ALTER SYSTEM SET db_files = 300 SCOPE=SPFILE;

-- Example: test SCOPE=MEMORY only (temporary)
ALTER SYSTEM SET open_cursors = 500 SCOPE=MEMORY;
Not all parameters support all scopes!
  • Static parameters (like processes) → only SCOPE=SPFILE works (the instance is running, memory can't be changed live)
  • Dynamic parameters (like open_cursors) → all 3 scopes are valid
  • If you try SCOPE=MEMORY on a static param, Oracle gives: ORA-02095: specified initialization parameter cannot be modified
Step 9
Revert all modifications made to the spfile.
SQL
-- Method 1: Reset a specific parameter to its default value
ALTER SYSTEM RESET processes SCOPE=SPFILE;
ALTER SYSTEM RESET open_cursors SCOPE=SPFILE;

-- Method 2: Recreate the SPFILE from a clean PFILE
CREATE SPFILE FROM PFILE='path/to/clean/init.ora';

-- After resetting static params, restart to apply:
SHUTDOWN IMMEDIATE;
STARTUP;
ALTER SYSTEM RESET removes the parameter entry from the SPFILE, forcing Oracle to use its built-in default value on next startup. This is the clean way to undo a SPFILE change without knowing the original value.
Exercise 2

Dynamic Performance Views

What are Dynamic Performance Views (v$ views)?

These are virtual views maintained by Oracle in memory. They give real-time information about the state of the database instance — memory usage, running processes, files, sessions, etc. They are called "dynamic" because the data changes continuously as the database operates. They do NOT persist data across shutdowns. They belong to SYS and start with V$ (or their synonyms GV$ for RAC).

Step 1
Display the logical block size defined by DB_BLOCK_SIZE.
SQL
SHOW PARAMETER db_block_size;
-- or:
SELECT value FROM v$parameter WHERE name = 'db_block_size';
db_block_size = size in bytes of one Oracle data block. Common values: 8192 (8KB, default). This is a static parameter — set at database creation and cannot be changed afterward. Larger blocks = fewer I/O operations for large sequential reads. Smaller blocks = better for OLTP random access.
Step 2
Display information about the SGA (System Global Area).
SQL
-- Summary of SGA components:
SELECT * FROM v$sga;

-- Detailed breakdown:
SELECT pool, name, bytes FROM v$sgastat ORDER BY pool, name;

-- Or using SHOW SGA:
SHOW SGA;
SGA (System Global Area) — memory shared by ALL Oracle processes:
  • Buffer Cache: Caches data blocks read from disk
  • Shared Pool: Caches SQL execution plans and data dictionary
  • Redo Log Buffer: Temporary buffer for redo log entries before writing to disk
  • Large Pool: Used for parallel queries, RMAN backups
  • Java Pool: Used for Java stored procedures
Step 3
Display information about the PGA (Program Global Area).
SQL
SELECT * FROM v$pgastat;
SGA vs PGA — key distinction for the exam:
SGAPGA
Stands forSystem Global AreaProgram Global Area
Shared byALL processesONE process (private)
ContainsShared cache, redo bufferSort area, session data
Viewv$sga, v$sgastatv$pgastat
Step 4
Display the list of options installed with the Oracle server.
SQL
SELECT parameter, value
FROM  v$option
ORDER BY parameter;
v$option shows which Oracle options/features are installed (e.g., Partitioning, OLAP, Spatial, Label Security). value = TRUE means the option is installed and licensed; FALSE means it is not available.
Step 5
Display the list of control files.
SQL
SELECT name, status, block_size
FROM  v$controlfile;
Control files are critical binary files that Oracle reads at startup. They contain: database name, location of datafiles and redo logs, checkpoint info, RMAN backup info. Oracle requires at least one control file to start. Best practice: maintain 2–3 copies on different disks.
Step 6
Display the current status of memory_target and memory_max_target parameters.
SQL
SELECT name, value, description
FROM  v$parameter
WHERE name IN ('memory_target', 'memory_max_target');
Automatic Memory Management (AMM):
  • memory_target → total memory Oracle can use for SGA+PGA combined. If set > 0, Oracle manages SGA/PGA automatically.
  • memory_max_target → the absolute ceiling. memory_target can never exceed this.
  • If both are 0, Oracle uses manual SGA/PGA sizing (via sga_target, pga_aggregate_target).
Step 7
Display the list of background processes.
SQL
-- Only show processes that are actually running (paddr != '00')
SELECT name, description
FROM  v$bgprocess
WHERE paddr != '00'
ORDER BY name;
Key Oracle background processes to know:

Step 8
Query v$DATABASE to display the name of the database used in the current instance.
SQL
SELECT name, db_unique_name, cdb, con_id,
       log_mode, open_mode, created
FROM  v$database;
v$database is one of the most important views. Key columns: name = database name, log_mode = ARCHIVELOG or NOARCHIVELOG, open_mode = READ WRITE / READ ONLY / MOUNTED, cdb = YES/NO (is it a Container DB?).
Step 9
Use: V$CONTROLFILE, V$DATAFILE, V$TABLESPACE, V$LOGFILE.
SQL
-- Control files (binary metadata files)
SELECT name, status FROM v$controlfile;

-- Data files (where actual table/index data is stored)
SELECT file#, name, status, bytes/1024/1024 size_mb
FROM  v$datafile;

-- Tablespaces (logical storage containers grouping datafiles)
SELECT ts#, name, included_in_database_backup
FROM  v$tablespace;

-- Redo log files (used for recovery)
SELECT group#, member, status, type
FROM  v$logfile;
Oracle physical file types — must know:
File TypeViewPurpose
Control Filev$controlfileDatabase structure metadata, required at startup
Data File (.dbf)v$datafileStores actual data (tables, indexes, etc.)
Redo Log Filev$logfileRecords every change for crash recovery
Parameter Filev$parameter / v$spparameterStartup configuration
Archive Logv$archived_logSaved redo logs for point-in-time recovery

Tablespace vs Datafile: A tablespace is logical (e.g., USERS, SYSTEM, TEMP). A datafile is physical (a .dbf file on disk). One tablespace can span multiple datafiles.
